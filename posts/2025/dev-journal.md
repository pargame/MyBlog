# 개발일지 1 — 지식 그래프 블로그를 세우기까지

## 서론
처음 요청은 단순했습니다. “개인 블로그를 만들고, 문서를 옵시디언처럼 그래프로 보고 싶다.” 그러나 실제 구현은 단순하지 않았습니다. 포스트와 아카이브의 역할을 분리하고, 보안과 가독성을 챙기고, 배포 자동화까지 신경 써야 했죠. 이 글은 그 과정을 요약하고, 무엇을 선택했고 왜 그렇게 했는지, 그리고 다음에 개선할 점을 정리한 개발자의 기록입니다.

## 본론
### 1) 콘텐츠 모델: 포스트 vs. 아카이브 분리
- posts/: 블로그 홈에 노출되는 “포스트” 전용 위치
- docs/: 지식 그래프에만 노출되는 “아카이브/노트” 위치(레거시)
- 빌더는 파일 경로(posts/ vs docs/)를 기준으로 역할을 구분하여, 홈과 그래프의 스코프가 섞이지 않도록 설계했습니다.

이 구분 덕분에 홈은 최신 글을, 그래프는 주제 간 연결을 보여주도록 명확히 분리되었습니다.

### 2) 그래프 빌드 파이프라인
- scripts/build-graph.js가 Markdown을 스캔합니다.
  - 위키링크 [[FileName]]을 파싱해 노드 간 엣지를 생성
  - Front matter에서 author, date를 추출
  - 폴더 이름과 tags를 주제 토픽으로 정규화
- 결과는 public/graph.json으로 출력됩니다.

이 데이터는 graph.html에서 D3.js로 시각화되고, viewer.html에서 역링크(backlinks) 탐색과 내부 링크 라우팅에 재사용됩니다.

### 3) 보안과 렌더링 품질
- Markdown 렌더러는 marked → DOMPurify → highlight.js 순으로 처리
- iframe 임베드 시 헤더 축약, 코드 블록 강조, 앵커 ID를 지원
- 내부 링크는 [[노트]] 문법을 클릭하면 동일 아카이브/토픽을 우선하는 경로로 변환해 맥락을 잃지 않게 했습니다.

### 4) UI/UX 다듬기
- 그래프 컨트롤(거리/반발/두께/크기)과 초기 배치 안정화
- 아카이브와 토픽 필터를 URL에 반영하여 공유 가능
- 홈은 최근 포스트 최대 20개만 노출하고, README/Index 류는 자동 제외

### 5) 배포와 메타
- GitHub Pages로 자동 배포; 중복 워크플로 제거
- scripts/build-meta.js가 public/meta.json(version, commit)을 생성
- footer는 저장소 정보를 읽어 표시

## 결론
이제 블로그는 다음을 만족합니다.
- 홈은 “포스트”만, 그래프는 “아카이브/노트”만 보여줍니다.
- 문서 간 연결은 위키링크로 자연스럽게 생성되고, 뷰어에서 안전하게 렌더링됩니다.
- 배포는 자동화되었고, 메타 정보로 버전/커밋을 추적합니다.

다음 스텝 제안
- 검색(제목/본문/태그) 추가
- 그래프에서 다중 토픽 교집합 필터
- 포스트 템플릿(서론-본론-결론) 자동 생성 스니펫

이 글은 첫 번째 개발일지입니다. 이후 변화도 같은 포맷으로 정리해, 기술적 의사결정의 맥락을 꾸준히 기록하겠습니다.
