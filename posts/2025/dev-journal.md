# 블로그 개발 일지

첫 배포를 해놓고 보니, “작동은 하는데 마음이 불편한” 구석이 계속 눈에 밟혔습니다. 아래는 그 불편함을 하나씩 추적해 고친 과정입니다. 실패와 우회, 그리고 작은 기쁨까지 남깁니다.

## 1. 홈에 아카이브 문서가 끼어들다
- 현상: 홈의 최근 글에 아카이브 문서, 심지어 README/index 류까지 보임.
- 원인: 포스트/문서 구분을 토픽(라벨)로 억지 판별했고, README 필터링이 느슨했음.
- 시도: 설정값(postsCollection)으로 선택 시도 → 일부 케이스 놓침.
- 해결: 파일 경로 기준으로 단일화(`node.file`이 `posts/`로 시작하면 포스트). 그리고 README/index 정규식 정확화.
- 교훈: “데이터 설계는 모호함을 남기지 말 것.” 경로는 가장 단순하고 강력한 시그널.

## 2. 그래프에 posts가 떠버리다
- 현상: 그래프에 ‘Posts’ 토픽만 달린 고립 노드가 생김.
- 원인: 빌더가 루트 라벨(Posts/Docs)을 토픽에 주입함.
- 해결: 루트 라벨을 토픽에 주입하지 않도록 빌더 수정. 그래프 쪽도 posts는 파일 경로로 필터.
- 교훈: 표현용 라벨을 데이터 모델에 섞어 넣으면, 결국 표시층에서 새는 구멍이 생긴다.

## 3. [[위키링크]]가 404를 낸다
- 현상: 포스트의 링크 클릭 시 `./2025/hello-world.md`를 요청하며 404.
- 원인: 뷰어가 id만 보고 상대 경로를 지어냈음(루트 정보 상실).
- 해결: 그래프 노드의 `file` 경로를 우선 사용해 정확한 파일을 로드. archive/topic 컨텍스트 우선순위를 유지해 해석.
- 교훈: 링크는 “보이는 텍스트”가 아니라 “정규화된 id → 파일 경로”로 연결되어야 견고하다.

## 4. Topics에 ‘Docs’가 왜 보여?
- 현상: Topics 영역에 ‘Docs’가 등장.
- 원인: (2)와 동일—루트 라벨을 토픽으로 취급.
- 해결: 빌더에서 주입 제거 + UI에서도 방어 필터. 이후엔 파일 경로 판별이 주 역할.
- 교훈: 한 번의 타협(루트 라벨 주입)은 두 번의 예외 처리(그래프/토픽)를 낳는다.

## 5. 제목이 [object HTMLHeadingElement]로 나온 날
- 현상: 뷰어 상단 제목이 객체 문자열로 출력.
- 원인: finalize에 실수로 DOM 요소가 흘러들어감.
- 해결: finalize(null)로 호출해 본문 H1에서만 추출. 추가로 frontmatter를 렌더 전에 제거.
- 교훈: “관심사의 분리.” 렌더-파싱-표시 경계를 명확히.

## 6. 그래프가 덜컹거리다(점프/충돌/슬라이더)
- 현상: 컨트롤 리셋/토픽 전환 시 그래프 높이 점프, 노드 충돌, 슬라이더 정렬 깨짐.
- 해결: 컨테이너 크기 캐싱, forceCollide 반경 동기화, 슬라이더 레이아웃을 그리드로 통일.
- 덤: 이웃 하이라이트/선택 테두리 UX를 깔끔하게 정리.

## 7. 이메일은 보여야 하고, 또 보이면 안 된다
- 현상: Footer에서 이메일을 안 가리고 싶고, 또 크롤링도 싫음.
- 해결: CSS `::before`로 전체 주소를 생성해 표시는 하되 DOM 텍스트에는 남기지 않음.
- 교훈: 단순하지만 효과적인 얕은 난독화도 사용자 경험-보안 사이 좋은 절충점.

## 8. 배포·자동화의 작은 비탈길
- 현상: 반복 커밋/푸시와 사소한 승인 대기.
- 해결: 워크스페이스 자동 승인/간소화 설정 + 스크립트 정리.
- 교훈: “한 번 번거로우면 스크립트로 두 번 편하게.”

## 오늘의 감정
여러 번 고개를 갸웃했고, 몇 번은 왜 이게 안 되지 싶었지만—문제를 작은 단위로 잘라 원인-증상을 연결하는 순간, 화면이 ‘제대로’ 보이기 시작했습니다. 그 짧은 순간이 좋아서, 다음 문제를 또 찾습니다.
